## Teil 3.9 — Page Object Pattern

Bisher kannte jeder Test die Seite selbst: `getByRole`, `getByLabel`, die
Struktur des Drawers, die Namen der Buttons. Solange ein Test das alleine tut,
geht das gut. Beim fünften Test, der denselben Bestellvorgang durchläuft, nicht
mehr.

Ein **Page Object** ist eine Klasse, die diese Kenntnis übernimmt: Sie kennt die
Seite, die Tests kennen nur noch die Fachsprache. Aus

```ts
await page
  .getByRole("article", { name: "Mojito" })
  .getByRole("button", { name: "In den Warenkorb" })
  .click();
```

wird

```ts
await bar.legeInDenWarenkorb("Mojito");
```

Der Gewinn ist nicht die kürzere Zeile. Der Gewinn ist, dass es **eine Stelle**
gibt, an der steht, wie man einen Cocktail in den Warenkorb legt.

### Die Ausgangslage

In `e2e/3.9-page-object.spec.ts` liegen zwei grüne Tests — Ablauf A und Ablauf B.
Sie tun in weiten Teilen dasselbe, und beide sagen es Locator für Locator.

Lies sie einmal durch und markiere alles, was doppelt dasteht. Besonders auffällig
ist der Block, der zur Bestellung wechselt: elf Zeilen, zweimal wörtlich. Er sieht
komplizierter aus, als er müsste — das hat einen Grund, der im Kommentar der
Methode `zurBestellung()` steht. Lies ihn.

### Die Anforderungen

Die Bar hat fünf Abläufe, die abgesichert sein sollen. A und B gibt es schon (als
Tests, die du umbaust), C bis E stehen als leere Rümpfe in der Spec-Datei und
sind neu zu schreiben.

**Ablauf A · Schnelle Runde**
Ein Gast sucht „Mojito", legt ihn in den Warenkorb und bestellt ihn auf den Namen
*Anna* an Tisch 7. Danach: Erfolgs-Toast, der Zähler steht auf 1.

**Ablauf B · Korrigierte Runde**
Ein Gast legt Mojito und Cuba Libre in den Warenkorb, erhöht die Menge des Mojito
auf 3 (die Position steht dann bei 25,50 €), entfernt den Cuba Libre wieder und
bestellt auf den Namen *Bosse*. Danach: Erfolgs-Toast, der Warenkorb ist leer.

**Ablauf C · Sonderwunsch**
Ein Gast öffnet den Konfigurator der Caipirinha (9,00 €), setzt die Stärke auf 2,
wählt „Brauner Zucker" ab und hinterlässt die Notiz „wenig Eis bitte". Der Preis
im Konfigurator steht danach bei 18,00 €. Er übernimmt, öffnet den Warenkorb — die
Position steht bei 18,00 € — und bestellt auf *Carla*.

**Ablauf D · Gutschein**
Ein Gast legt eine Virgin Colada in den Warenkorb, erhöht die Menge im Drawer auf 2
(Position: 13,00 €), geht zur Bestellung, löst den Gutschein „HAPPY" ein — der
Dialog meldet „Rabatt: 15 %" — und bestellt auf *Dana* mit 2 € Trinkgeld.

**Ablauf E · Alkoholfreie Runde**
Ein Gast setzt den Filter „nur alkoholfrei" (es bleiben zwei Karten übrig), legt
beide in den Warenkorb, öffnet ihn — zwei Positionen — und bestellt auf *Elif* an
Tisch 3. Danach ist der Warenkorb wieder leer.

### Die Aufgabe

**Schritt 1 — Bestandsaufnahme.** Schreib dir die Schritte auf, die in den fünf
Abläufen vorkommen. Nicht die Locators, die *Handlungen*: suchen, filtern, in den
Warenkorb legen, konfigurieren, Menge ändern, entfernen, zur Bestellung gehen,
Gutschein einlösen, bestellen. Welche kommen in wie vielen Abläufen vor?

**Schritt 2 — Methoden schneiden.** Vervollständige die Klasse `HappyHourSeite`.
Der Konstruktor, `oeffnen()`, `karte()` und `zurBestellung()` sind schon da; im
`TODO`-Block stehen Namensvorschläge für den Rest.

Drei Regeln:

- **Ein Schritt = eine Methode.** Was ein Test einzeln braucht, darf nicht in einer
  größeren Methode eingesperrt sein.
- **Assertions gehören in den Test.** Das Page Object liefert Locators
  (`warenkorbPosition("Mojito")`), der Test entscheidet, was daran erwartet wird.
  Ein Page Object, das selbst prüft, versteckt die Erwartung vor dem Leser.
- **Kein `page.getByRole(...)` im Testkörper.** Taucht dort einer auf, fehlt eine
  Methode.

**Schritt 3 — A und B umstellen.** Die Assertions bleiben unverändert, nur die
Wege dorthin wandern in die Klasse. Beide Tests müssen weiter grün sein — du
refactorst, du änderst nicht, was geprüft wird.

**Schritt 4 — C, D und E schreiben.** Entferne das `test.fixme` und schreib die
Abläufe. Wenn dir dabei eine Methode fehlt, ergänze sie — das ist der Normalfall,
nicht das Versagen.

**Schritt 5 — vorlesen.** Lies deine fünf Tests laut. Wenn sich einer wie eine
Geschichte anhört, die auch der Barkeeper verstehen würde, ist der Schnitt gut.

### Überlegungen

Sobald du fertig bist, versuch, folgende Methode zu schreiben:

```ts
async bestelleCocktail(cocktail: string, gast: string) { … }
```

Eine Methode, die alles macht: hinzufügen, Warenkorb öffnen, bestellen, absenden.
Wie viele deiner fünf Abläufe kannst du damit tatsächlich abdecken?

Ablauf B muss zwischen „in den Warenkorb" und „bestellen" die Menge ändern.
Ablauf D muss zwischen „zur Bestellung" und „absenden" den Gutschein einlösen.
Ablauf C schiebt eine ganze Konfiguration dazwischen. Die Mega-Methode passt auf
genau einen Ablauf — auf A. Genau deshalb schneidet man Page Objects an den
**Schritten** der Anwendung, nicht an den **Abläufen** der Tests.

Zwei weitere Fragen:

Der `toPass`-Block aus `zurBestellung()` stand vorher zweimal im File. Jetzt steht
er einmal. Was passiert, wenn die App den Hydration-Zeitpunkt verschiebt — und was
wäre passiert, wenn er noch in fünf Tests klebte?

Was passiert, wenn die Bar den Button von „In den Warenkorb" in „Aufs Tablett"
umbenennt? Zähl die Stellen, die du vorher anfassen müsstest, und die Stellen jetzt.

### Fallstricke

Das Muster lässt sich auch übertreiben:

- **Das Gott-Objekt.** Eine Klasse für die ganze App. Bei fünf Seiten hat sie
  200 Methoden. Ein Page Object pro Seite oder pro größerem Bereich.
- **Assertions im Page Object.** `async pruefeToast()` klingt praktisch, versteckt
  aber die Erwartung. Der Test soll zeigen, was gilt.
- **Vererbungstürme.** `BasePage` → `AbstractDialogPage` → `WarenkorbPage`. Meist
  reicht Komposition; die Klasse ist ein Werkzeug, kein Framework.
- **Zu dünne Wrapper.** `async klickeButton(name)` benennt nichts um, sondern
  verschiebt nur `getByRole`. Methoden tragen Namen aus der **Domäne**, nicht aus
  dem DOM.

### Bonus

- Zieh die Klasse in eine eigene Datei (`e2e/pages/happy-hour-seite.ts`) und
  importiere sie. So sieht es in echten Projekten aus.
- Bau daraus eine **Fixture**, damit `bar` direkt als Test-Argument ankommt und das
  `beforeEach` verschwindet:

  ```ts
  const test = base.extend<{ bar: HappyHourSeite }>({
    bar: async ({ page }, use) => {
      const bar = new HappyHourSeite(page);
      await bar.oeffnen();
      await use(bar);
    },
  });

  test("Ablauf A …", async ({ bar }) => { … });
  ```

- Schreib einen sechsten Ablauf, ohne die Klasse anzufassen. Wenn das gelingt, war
  der Schnitt richtig.

Die Lösung liegt in `e2e/3.9-page-object.solution.spec.ts` — schau erst rein, wenn
deine fünf Abläufe grün sind. Es gibt mehr als einen guten Schnitt.
